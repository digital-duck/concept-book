Web Application Backend

Every time you search for a product on an e-commerce site, something invisible handles your request: it looks up your account, queries a database of millions of items, applies filters, and returns a personalised list — all within milliseconds. That invisible layer is the backend.

Definition. A web application backend is the server-side component of a web system responsible for business logic, data persistence, and communication between clients and databases. Formally, we can model it as a function

$$B: (R, S) \rightarrow (R', S')$$

where $R$ is an incoming HTTP request, $S$ is the current application state (database contents, session data), $R'$ is the HTTP response returned to the client, and $S'$ is the updated state after the request is processed.

Worked Example. Consider a simple login endpoint. A client sends a POST request containing credentials $\{u, p\}$. The backend:

  • Receives $R = \texttt{POST /login}$ with body $\{u, p\}$.
  • Queries the database: $\texttt{SELECT hash FROM users WHERE username} = u$.
  • Verifies: $\texttt{verify}(p,\, \texttt{hash}) \in \{\text{true}, \text{false}\}$.
  • If true, creates a session token $\tau$ and returns $R' = 200\,\texttt{OK}$ with $\{\tau\}$; otherwise returns $R' = 401\,\texttt{Unauthorized}$.
  • State transitions: $S' = S \cup \{(\tau, u)\}$ if login succeeds, else $S' = S$.

    • Why it works. The mapping $B$ must be deterministic for identical inputs (same credentials always produce the same verification outcome) yet stateful (the token $\tau$ persists across subsequent requests). This separation of pure logic from mutable state is what makes backends testable and scalable independently of the frontend.

    Practice Problem. Design the state-transition function $B$ for a /logout endpoint. Define what $R$, $S$, $R'$, and $S'$ should be, and explain what invariant must hold about $S'$ after a successful logout.
